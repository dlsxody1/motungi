# 디자인 프로세스

사용자 플로우와 화면 디자인을 위한 단계별 가이드. 다음 단계를 순서대로 따르세요. 건너뛰거나 순서를 바꾸지 마세요.

---

## Step 1: 리서치 및 정의

어떤 디자인 작업이든, 누구를 위해 디자인하는지, 그들이 무엇을 필요로 하는지, 어떤 제약이 있는지 먼저 파악하세요.

### 1a. 사용자 의도
- 사용자가 무엇을 달성하려고 하는가? (당신의 기능이 아닌, 그들의 목표)
- 그들을 여기로 오게 한 트리거는 무엇인가? (진입점 컨텍스트: 알림, 검색, 링크, 습관)
- 그들의 관점에서 성공이란 어떤 모습인가? (완료가 아닌 — 만족)

### 1b. 사용자 컨텍스트
- 그들은 어떤 환경에 있는가? (출퇴근 중, 업무 중, 휴식 중, 긴급한가?)
- 그들의 정신 상태는? (집중, 산만, 스트레스, 즐거움?)
- 어떤 디바이스와 플랫폼? (모바일 한손, 태블릿, 키보드 있는 데스크톱?)
- 시간 압박은 어떤가? (빠른 작업 vs 깊은 작업 세션?)

### 1c. 가벼운 리서치 방법

배워야 할 것에 따라 선택하세요.

| 질문 | 방법 | 시간 |
|------|------|------|
| 사용자는 누구인가? | **프로토 페르소나** 스케치 | 30분 |
| 사용자는 무엇이 필요한가? | **사용자 인터뷰** (5명) | 1-2일 |
| 그들은 이것을 어떻게 생각하는가? | **카드 정렬** (15명 이상) | 1일 |
| 그들이 하려는 것은 무엇인가? | **Jobs-to-be-Done** 프레이밍 | 1시간 |
| 오늘 어디서 어려움을 겪는가? | 현재 플로우의 **여정 맵** | 2-4시간 |

#### 프로토 페르소나 (간이형)
완전한 리서치를 할 수 없을 때, 가벼운 페르소나를 만드세요.
```
이름:        [현실적인 이름]
역할:        [그들이 하는 일]
목표:        [달성하려는 것]
컨텍스트:    [언제/어디서/어떻게 사용할지]
좌절 지점:   [현재의 페인 포인트]
기술 친숙도: [낮음 / 중간 / 높음]
```
규칙: 주요 플로우당 하나의 페르소나. 한 기능에 3개 이상의 별개 페르소나가 있다면, 그 기능이 너무 많은 일을 하려는 것입니다.

#### Jobs-to-be-Done (JTBD)
모든 기능을 사용자가 제품을 고용해서 시키는 일로 프레이밍하세요.
```
[상황]일 때, [동기]를 하고 싶다, 그래서 [기대 결과]를 얻을 수 있도록.

예시:
출퇴근 중에 회의 초대장을 받았을 때, 빠르게 가능 여부를 확인하고 싶다, 그래서 잊기 전에 응답할 수 있도록.
```
이 프레이밍은 기능의 비대화를 막습니다 — 모든 요소는 그 일을 수행하거나 하지 않거나입니다.

#### 여정 맵 (현재 상태)
페인 포인트를 찾기 위해 사용자의 현재 경험을 매핑하세요.
```
단계:    인지 → 고려 → 행동 → 행동 후
하는 일:  [각 단계에서 하는 일]
생각:    [궁금해하는 것]
감정:    [감정 상태: 자신감, 혼란, 불안, 안도]
페인:    [마찰이나 좌절이 발생하는 지점 — 이것이 디자인 기회]
```

### 1d. 정보 요구사항
- 각 단계에서 사용자가 진행하기 위해 필요한 최소 정보는 무엇인가?
- 무엇을 나중 단계로 미룰 수 있는가?
- 우리가 만들 수 있는 (그리고 자동 입력할 수 있는) 가정은 무엇인가?
- 사용자의 어휘는 무엇인가? (내부 전문 용어가 아닌 그들의 단어를 사용하라)

## Step 2: 핵심 경로 매핑

### 2a. 가장 짧은 경로 찾기
1. 진입에서 목표 완료까지 모든 단계를 나열
2. 각 단계에 대해 묻기: "이 단계가 사용자의 목표에 필수적인가?"
3. 가능한 경우 단계를 제거하거나 결합
4. 백그라운드에서 일어날 수 있는 단계 식별 (자동 저장, 비동기 처리)
5. 남은 단계 세기 — 5개 이상이라면, 더 제거할 것을 찾기

### 2b. 해피 패스를 먼저 디자인하라
- 80%의 사용자는 마찰 없는 경험을 가져야 한다
- 엣지 케이스는 처리하되, 메인 플로우의 비용으로는 안 됨
- 에러 상태는 해피 패스가 단단해진 후에 디자인
- 점진적 복잡성: 단순한 것 먼저, 고급 기능은 발견 가능하게

### 2c. 의사결정 지점 매핑
```
진입 → [결정 A] → 액션 1 → [결정 B] → 액션 2 → 성공
              ↓                          ↓
         대안 경로 1                대안 경로 2
```

각 의사결정 지점에 대해:
- 사용자가 결정하기 위해 어떤 정보가 필요한가?
- 옵션을 줄일 수 있는가? (`references/cognitive-principles.md`의 힉의 법칙 참조)
- 기본값을 추천할 수 있는가? (최선의 옵션을 미리 선택)
- 사용자가 결정하지 않으면 어떻게 되는가? (타임아웃, 기본값, 차단?)

### 2d. 상태 전환 매핑
각 화면에 대해, 가능한 모든 상태를 정의하세요.
```
[빈 상태] → [로딩] → [로드됨] → [에러]
                         ↕
                     [새로고침]
                         ↕
                     [부분]
                         ↕
                     [오프라인]
```
7가지 보편적 화면 상태는 `references/interaction-patterns.md` 참조.

### 2e. 정보 구조
여러 화면이 있는 기능에 대해:
- 내비게이션 패턴 정의 (탭, 스택, 모달, 드로어)
- 화면 위계 매핑 (사이트맵)
- 검증: 사용자가 ≤3 탭 안에 핵심 콘텐츠에 도달할 수 있는가?
패턴은 `references/information-architecture.md` 참조.

## Step 3: 각 화면 디자인

### 3a. 화면 해부도
```
┌─────────────────────────────┐
│  컨텍스트 (나는 어디에 있는가?)│
├─────────────────────────────┤
│                             │
│  정보                       │
│  (무엇을 알아야 하는가?)     │
│                             │
├─────────────────────────────┤
│  주요 액션                   │
│  (무엇을 해야 하는가?)       │
├─────────────────────────────┤
│  보조 액션                   │
│  (또 무엇을 할 수 있는가?)   │
└─────────────────────────────┘
```

### 3b. 화면별 명세

각 화면에 대해 정의하세요.

1. **정체성**: 화면 이름, URL/라우트, 진입점
2. **주요 액션** (하나 — 시각적으로 우세)
3. **표시 정보** (현재 의사결정을 지원하는 것만)
4. **보조 액션** (사용 가능하지만 주요 액션과 경쟁하지 않음)
5. **피드백 메커니즘** (주요 액션 후 어떤 일이 일어나는가)
6. **모든 상태** (빈 상태, 로딩, 로드됨, 에러, 부분, 새로고침, 오프라인)
7. **출구 경로** (사용자가 여기서 어디로 갈 수 있는가)

### 3c. 콘텐츠 위계
1. 사용자가 가장 먼저 필요한 것은 무엇인가? → 화면 상단 / 폴드 위
2. 주요 의사결정을 지원하는 것은 무엇인가? → 중간
3. 무엇이 선택적/보조인가? → 하단 또는 점진적 공개

### 3d. 인터랙션 명세

각 인터랙티브 요소에 대해 정의하세요.
- **트리거**: 탭, 스와이프, 롱 프레스, 호버
- **응답**: 시각적, 기능적으로 무엇이 일어나는가
- **피드백**: 즉각적인 시각 변화 + 햅틱 (모바일인 경우)
- **에러 케이스**: 액션이 실패하면 어떻게 되는가
표준 패턴은 `references/interaction-patterns.md` 참조.

### 3e. UX 카피

각 화면에 대해 명세하세요.
- **헤딩과 라벨** (명확하고, 액션 지향)
- **버튼 라벨** (구체적인 동사 — `references/ux-writing.md` 참조)
- **빈 상태 카피** (도움이 되는 것 — 단지 "여기 아무것도 없음"이 아닌)
- **에러 메시지** (무엇이 일어났는지 + 어떻게 고칠지)
- **헬퍼 텍스트** (필드 입력 형식이 명확하지 않다면)

## Step 4: 제거

초기 디자인 후, 공격적으로 잘라내세요. 이것이 가장 중요한 단계입니다.

### 4a. 제거 테스트
모든 화면의 모든 요소에 대해 묻기:
> "이것을 제거하면, 사용자가 목표를 완료하는 것을 막는가?"
> - 예 → 유지
> - 아니오 → 제거
> - 모르겠음 → 테스트 대상으로 표시

### 4b. 제거 체크리스트

| 자주 추가되는 것 | 제거 조건... |
|-----------------|-------------|
| 튜토리얼/온보딩 | UI가 자명한 경우 |
| 확인 다이얼로그 | 액션이 비파괴적이거나 되돌릴 수 있는 경우 |
| "정말 하시겠습니까?" 프롬프트 | 액션이 실행 취소될 수 있는 경우 |
| 기능 설명 | 라벨이 명확한 경우 |
| 제품 안의 마케팅 카피 | 사용자가 작업 중인 경우 |
| 소셜 증거/배지 | 현재 의사결정에 도움이 안 되는 경우 |
| 초기에 노출된 설정 | 기본값이 80% 이상 사용자에게 작동하는 경우 |
| 화면당 여러 CTA | 그들이 경쟁하여 모든 클릭률을 떨어뜨리는 경우 |
| 장식적 일러스트 | 유용한 정보를 전달하지 않는 경우 |

### 4c. 단계 축소
- 어떤 두 단계를 하나로 결합할 수 있는가?
- 어떤 단계를 자동화할 수 있는가 (위치 자동 감지, 자동 입력, 자동 저장)?
- 어떤 단계를 나중으로 미룰 수 있는가 ("혹시 모르니"가 아닌 실제로 필요할 때 정보 수집)?
- 어떤 결정을 스마트 기본값으로 만들 수 있는가?

### 4d. 안티 패턴 점검
`SKILL.md`의 안티 패턴 목록과 교차 참조하세요. 어떤 요소가 안티 패턴과 일치한다면, 제거하거나 재디자인하세요.

## Step 5: 검증

### 5a. 셀프 리뷰 워크스루
신규 사용자인 척 전체 플로우를 걸어보세요.
1. 각 단계에서 무엇을 해야 할지 알겠는가? (모호함 없음)
2. 헷갈리는 게 있는가? (표시)
3. 시작에서 목표까지 몇 번의 탭/클릭이 필요한가? (세어보기)
4. 무엇이 잘못될 수 있는가? (실패 모드 나열)
5. 모든 상태가 존재하는가? (빈 상태, 로딩, 에러 — 해피 패스만이 아닌)

### 5b. 체크리스트 검증
모든 화면에 대해 다음을 실행:
- [ ] `SKILL.md`의 빠른 체크리스트
- [ ] 인지 원칙 (원칙 이름으로 정당화됨)
- [ ] 인체공학 사양 (`references/ergonomics.md`)
- [ ] UX 라이팅 규칙 (`references/ux-writing.md`)
- [ ] 인터랙션 패턴 (`references/interaction-patterns.md`)
- [ ] IA 원칙 (`references/information-architecture.md`)

### 5c. 5명 사용자 사용성 테스트 프로토콜

5명의 사용자가 사용성 이슈의 약 85%를 발견합니다 (Nielsen, 2000). 프로토콜은 다음과 같습니다.

#### 셋업
- **참가자**: 프로토 페르소나에 일치하는 5명
- **태스크**: JTBD 진술에 일치하는 3-5개의 핵심 태스크
- **형식**: 생각을 말하기 프로토콜 (사용자가 자기 생각을 내레이션)
- **시간**: 세션당 20-30분

#### 스크립트 템플릿
```
1. 인트로 (2분):
   "저는 디자인을 테스트하는 거지, 당신을 테스트하는 게 아닙니다. 정답은 없습니다.
    소리내어 생각해 주세요 — 무엇을 보고 있고 무엇을 할지 말씀해 주세요."

2. 태스크 (각각 반복):
   "당신이 [JTBD 시나리오]를 하고 싶다고 상상해 보세요.
    이 화면부터 시작해서, 어떻게 할지 보여주세요."

   관찰. 안내하지 않기. 메모할 것:
   - 어디를 먼저 탭하는지 (예상한 곳? 예상치 못한 곳?)
   - 어디서 망설이는지 (>3초 = 마찰)
   - 어디서 되돌아가는지 (잘못된 경로를 갔음)
   - 무엇을 말하는지 ("이게 무슨 뜻인지 모르겠어요")

3. 태스크 후 (태스크당 1분):
   "어떠셨어요? 헷갈렸던 게 있다면 무엇인가요?"

4. 마무리 (2분):
   "전반적으로, 좋든 나쁘든 두드러진 것은 무엇이었나요?"
```

#### 분석
| 발견 사항 | 심각도 | 빈도 (5명 중) | 조치 |
|---------|--------|--------------|------|
| [이슈] | 심각 / 주요 / 사소 | X/5 | [수정 / 조사 / 보류] |

심각도 정의:
- **심각**: 사용자가 태스크를 완료할 수 없음 (출시 전에 수정)
- **주요**: 사용자가 상당한 어려움으로 완료 (이번 이터레이션에서 수정)
- **사소**: 사용자가 알아채지만 우회함 (다음 이터레이션에서 수정)

### 5d. 출시 후 추적할 지표

| 지표 | 무엇을 나타내는가 | 적신호 임계값 |
|------|----------------|--------------|
| 완료율 | 플로우 명료성 | 핵심 태스크에 대해 <70% |
| 태스크 소요 시간 | 효율성 | 예상 시간의 >2배 |
| 이탈 지점 | 마찰 위치 | 어떤 단계에서든 >30% 이탈 |
| 에러율 | 혼란 | 사용자의 >10%가 에러를 만남 |
| 분노 탭/클릭 | 좌절 | 어떤 발생이든 |
| 재방문율 | 가치 전달 | 7일 이내 <30% 재방문 |
