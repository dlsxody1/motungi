# 인터랙션 패턴

화면이 사용자 액션에 어떻게 응답하는가. 인터랙션 디자인은 정적 레이아웃과 살아있는 인터페이스 사이의 다리입니다.

---

## 핵심 원칙

> 모든 사용자 액션은 가시적이고, 즉각적이고, 비례적인 응답을 만들어야 한다. 침묵은 자신감의 적이다.

사용자가 무언가를 했는데 아무 일도 일어나지 않으면, 그들은 액션을 반복하거나(에러를 만듦) 플로우를 이탈합니다(드롭오프를 만듦).

---

## 상태 머신 디자인

모든 인터랙티브 요소와 모든 화면은 유한한 상태들 중 하나에 존재합니다. "해피" 상태를 디자인하기 전에 모든 상태를 디자인하세요.

### 7가지 보편적 화면 상태

```
┌─────────┐     ┌─────────┐     ┌─────────┐
│ 빈 상태 │────▶│  로딩   │────▶│ 로드됨  │
└─────────┘     └─────────┘     └────┬────┘
                     │               │
                     ▼               ▼
                ┌─────────┐     ┌─────────┐
                │  에러   │     │  부분   │
                └─────────┘     └─────────┘
                                     │
                                     ▼
                                ┌─────────┐
                                │ 새로고침 │
                                └─────────┘
```

| 상태 | 사용자가 보는 것 | 디자인 요구사항 |
|------|----------------|----------------|
| **빈 상태** | 아직 콘텐츠 없음 | 왜 비어있는지 설명 + 첫 항목을 만들/추가할 명확한 CTA. 절대 공백이 아님. |
| **로딩** | 콘텐츠 가져오는 중 | <3초에는 스켈레톤 시머. >3초에는 진행 바. 컨텍스트: 무엇이 로드 중인지. |
| **로드됨** | 콘텐츠 표시됨 | 주요 콘텐츠 가시. 액션 사용 가능. |
| **에러** | 무언가 실패 | 무엇이 잘못됐는지 + 무엇을 할지 (재시도, 입력 변경, 지원 문의). 절대 사용자를 탓하지 마라. |
| **부분** | 일부 콘텐츠 로드, 일부 실패 | 성공한 것을 표시. 실패한 것에 인라인 에러. 모든 것을 차단하지 마라. |
| **새로고침** | 콘텐츠 업데이트 중 | 미묘한 인디케이터 (당겨서 새로고침 스피너). 가시 콘텐츠를 스켈레톤으로 대체하지 마라. |
| **오프라인** | 네트워크 연결 없음 | 캐시된 콘텐츠가 있으면 표시. 액션을 동기화 큐에. 배너: "오프라인 상태 — 변경사항은 연결 시 동기화됩니다." |

### 상태 체크리스트
어떤 화면이든 확정하기 전에:
- [ ] 7가지 상태 모두 디자인됨 (또는 어떤 상태가 불가능한지 명시적으로 문서화)
- [ ] 빈 상태에 실행 가능한 가이드가 있음
- [ ] 에러 상태에 복구 경로가 있음
- [ ] 로딩 상태가 예상 대기 시간과 일치
- [ ] 상태 간 전환이 매끄러움 (레이아웃 시프트 없음)

## 시스템 피드백 패턴

### 피드백 선택 가이드

| 시나리오 | 패턴 | 지속시간 | UI 차단? |
|---------|------|---------|---------|
| 액션 성공 (비핵심) | 토스트 / 스낵바 | 3-5초 자동 닫힘 | 아니오 |
| 액션 성공 (중요) | 인라인 확인 | 다음 액션까지 지속 | 아니오 |
| 실행 취소 윈도우 필요 액션 | 실행 취소 있는 스낵바 | 5-8초 | 아니오 |
| 파괴적 액션 확인 | 다이얼로그 / 알림 | 사용자 응답까지 | 예 |
| 검증 에러 | 인라인 에러 | 수정까지 | 아니오 |
| 시스템 레벨 이슈 | 배너 | 해결까지 | 아니오 |
| 백그라운드 프로세스 완료 | 푸시 알림 / 배지 | 확인까지 | 아니오 |

### 토스트 / 스낵바

```
┌─────────────────────────────────┐
│                                 │
│         화면 콘텐츠              │
│                                 │
│                                 │
├─────────────────────────────────┤
│ ✓ 항목 저장됨           [실행 취소]│  ← 스낵바
└─────────────────────────────────┘
```

규칙:
- 위치: 화면 하단, 탭 바 위 (모바일) 또는 좌하단 (웹)
- 자동 닫힘: 3-5초
- 한 번에 하나 (여러 개라면 큐에)
- 되돌릴 수 있는 작업에는 실행 취소 액션 포함
- 에러에는 절대 사용 금지 (에러는 지속적 가시성 필요)

### 인라인 피드백

```
┌─────────────────────────────────┐
│ 이메일                          │
│ ┌─────────────────────────────┐ │
│ │ notanemail                  │ │
│ └─────────────────────────────┘ │
│ ⚠ 유효한 이메일 주소를 입력하세요 │  ← 인라인 에러
│                                 │
│ 이름                            │
│ ┌─────────────────────────────┐ │
│ │ 홍길동                  ✓   │ │  ← 인라인 성공
│ └─────────────────────────────┘ │
└─────────────────────────────────┘
```

규칙:
- 별도 영역이 아닌 입력 필드 아래에 표시
- 블러 시 검증 (모든 키 입력 시가 아닌 — 불안 감소)
- 에러: 빨간 텍스트 + 아이콘. 성공: 초록 체크마크 (선택)
- 에러 메시지 형식: 무엇이 잘못됐는지 + 어떻게 고칠지
- 에러 시 사용자의 입력을 지우지 마라

### 확인 다이얼로그 (파괴적 액션만)

```
┌─────────────────────────────────┐
│                                 │
│  이 프로젝트를 삭제하시겠습니까? │
│                                 │
│  "내 프로젝트"와 모든 콘텐츠가  │
│  영구적으로 제거됩니다. 이      │
│  작업은 되돌릴 수 없습니다.     │
│                                 │
│  [취소]            [삭제]       │
│                                 │
└─────────────────────────────────┘
```

규칙:
- 파괴적 + 비가역 액션에만 사용
- 제목: "정말 하시겠습니까?"가 아닌 수행될 구체적인 액션
- 본문: 무엇이 일어날지 + 결과
- 주요 버튼: 파괴적 액션 (빨강), 구체적 동사로 라벨 ("OK"가 아닌 "삭제")
- 보조 버튼: 취소 (항상 탈출 경로)
- 사용 금지: 저장, 닫기 (데이터 손실 없이), 내비게이션

## 제스처 패턴

### 표준 제스처 어휘

| 제스처 | 표준 의미 | 플랫폼 노트 |
|-------|---------|------------|
| 탭 | 선택 / 활성화 | 보편적 |
| 롱 프레스 | 컨텍스트 메뉴 / 보조 액션 | iOS: 핍. Android: 컨텍스트 메뉴 |
| 좌/우 스와이프 | 삭제, 보관, 리스트 항목의 액션 | 아래에 액션 버튼 노출 |
| 아래로 스와이프 | 당겨서 새로고침 (스크롤 가능 리스트) | 놓으면 스프링 애니메이션 |
| 가장자리 스와이프 (왼쪽 가장자리 →) | 뒤로 가기 | iOS 시스템 제스처. 오버라이드 금지. |
| 핀치 | 줌 인/아웃 | 지도, 이미지, 문서 |
| 두 손가락 스크롤 | 콘텐츠 스크롤 | 표준 스크롤 동작 |
| 더블 탭 | 줌 핏 / 좋아요 (소셜) | 컨텍스트 의존 |

### 제스처 디자인 규칙

1. **새 제스처를 발명하지 마라** — 확립된 어휘 사용
2. **시스템 제스처를 오버라이드하지 마라** — iOS 가장자리 스와이프 백은 신성하다
3. **항상 가시적 대안 제공** — 제스처는 보이지 않고, 버튼은 보인다. 모든 제스처 액션은 가시적 컨트롤로도 사용 가능해야 한다
4. **햅틱 피드백 제공** — 스와이프 임계값에서 가벼운 햅틱, 액션 커밋에서 중간 햅틱
5. **스와이프 액션은 발견 가능해야 한다** — 첫 만남이나 온보딩에서 힌트 표시

### 스와이프-투-액션 패턴

```
일반 상태:
┌─────────────────────────────────┐
│ 📧 앨리스로부터의 이메일        │
│    내일 오후 3시 미팅           │
└─────────────────────────────────┘

왼쪽으로 스와이프 (부분):
┌──────────────────────┬──────────┐
│ 📧 앨리스로부터의 이메일│ 🗑 삭제 │
│    내일 미팅          │          │
└──────────────────────┴──────────┘

왼쪽으로 스와이프 (전체 — 파괴적):
┌─────────────────────────────────┐
│ 🗑 삭제됨              [실행 취소] │
└─────────────────────────────────┘
```

규칙:
- 부분 스와이프: 액션 노출 (보관, 삭제, 플래그)
- 전체 스와이프: 주요 액션 커밋
- 파괴적 스와이프 액션에는 항상 실행 취소 제공
- 색상 코딩: 빨강 = 파괴적, 초록/파랑 = 건설적
- 측면당 최대 3개 액션 (좌우)

## 낙관적 UI

액션이 성공한 것처럼 UI를 즉시 업데이트하고, 그 다음 서버 응답과 조정합니다.

### 사용 시점
- 액션의 성공률이 >95%
- 액션이 되돌릴 수 있음
- 실패가 데이터 손상을 일으키지 않음
- 예시: 게시물 좋아요, 읽음 표시, 설정 토글

### 사용 금지 시점
- 결제 처리
- 다른 사람에게 메시지 보내기
- 파괴적 액션 (삭제, 제거)
- 다른 사용자에게 보이는 부수 효과가 있는 액션

### 패턴

```
1. 사용자가 "좋아요" 탭
2. UI가 즉시 표시: 하트 채워짐, 카운트 +1
3. API 호출이 백그라운드에서 발화
4. 성공 시: 완료 (UI는 이미 정확)
5. 실패 시: UI 되돌리기 + 미묘한 에러 토스트 표시
```

## 로딩 패턴

### 대기 시간별 패턴 선택

| 예상 대기 | 패턴 | 예시 |
|----------|------|------|
| <300ms | 인디케이터 없음 | 버튼 상태 변경 |
| 300ms-2s | 스켈레톤 화면 | 레이아웃과 일치하는 콘텐츠 플레이스홀더 |
| 2s-5s | 스켈레톤 + 시머 애니메이션 | 피드 로딩 |
| 5s-15s | 진행 바 + 컨텍스트 | "사진 업로드 중..." with % |
| 15s+ | 백그라운드 프로세스 + 알림 | "준비되면 알려드릴게요" |

### 스켈레톤 화면

```
┌─────────────────────────────────┐
│ ████████████                    │  ← 제목 플레이스홀더
│ ██████████████████████████      │  ← 부제목 플레이스홀더
│                                 │
│ ┌────┐ ████████████████████     │  ← 아바타 + 텍스트
│ └────┘ ██████████████           │
│                                 │
│ ┌────┐ ████████████████████     │
│ └────┘ ██████████████           │
└─────────────────────────────────┘
```

규칙:
- 실제 콘텐츠 레이아웃과 일치 (일반적인 스피너가 아닌)
- 시머로 애니메이션 (좌우 펄스)
- 레이아웃 시프트 없이 실제 콘텐츠로 전환
- 캐시에 이미 있는 콘텐츠에는 스켈레톤 표시 금지

## 실행 취소 / 다시 실행 패턴

### 실행 취소 전략 선택

| 액션 유형 | 전략 | 예시 |
|----------|------|------|
| 되돌릴 수 있음, 낮은 위험 | 스낵바를 통한 실행 취소 (5-8초 윈도우) | 이메일 보관, 리스트에서 제거 |
| 되돌릴 수 있음, 중간 위험 | "최근 삭제됨"을 통한 실행 취소 (30일) | 사진 삭제, 노트 삭제 |
| 되돌릴 수 없음, 높은 위험 | 액션 전 확인 다이얼로그 | 계정 삭제, 결제 보내기 |
| 연속 편집 | Cmd+Z / Ctrl+Z 무제한 실행 취소 스택 | 텍스트 편집, 그리기, 디자인 |

### 소프트 삭제 패턴
```
사용자가 항목 "삭제"
→ 항목이 "최근 삭제됨"으로 이동 (메인 뷰에서 숨김)
→ 영구 삭제 전 30일 보관
→ 사용자가 윈도우 내 언제든 복구 가능
→ 30일 후: 영구 삭제 (확인 불필요 — 사용자에게 30일이 있었음)
```

## 전환 및 내비게이션 애니메이션

### 전환 선택

| 내비게이션 액션 | 애니메이션 | 지속시간 |
|---------------|----------|---------|
| 상세로 푸시 | 오른쪽에서 슬라이드 인 | 300ms |
| 리스트로 팝 백 | 오른쪽으로 슬라이드 아웃 | 250ms |
| 모달 열기 | 하단에서 슬라이드 업 | 300ms |
| 모달 닫기 | 슬라이드 다운 | 250ms |
| 탭 전환 | 크로스 페이드 (슬라이드 없음) | 200ms |
| 펼침/접힘 | 높이 애니메이션 + 페이드 | 200-300ms |

### 규칙
- 방향이 위계를 나타냄: 오른쪽 = 더 깊이, 왼쪽 = 뒤로, 위 = 오버레이, 아래 = 닫기
- 일치하는 전환: 열기와 닫기는 서로의 역이어야 함
- `prefers-reduced-motion` 준수: 애니메이션을 즉각 컷으로 대체
- 내비게이션에 바운시/스프링 애니메이션 사용 금지 (즐거움의 순간에만)
- 시각적 연속성을 위한 공유 요소 전환 (리스트 썸네일 → 상세 히어로 이미지)

## 마이크로 인터랙션

피드백과 즐거움을 제공하는 작고 격리된 애니메이션.

### 필수 마이크로 인터랙션

| 트리거 | 응답 | 목적 |
|-------|------|------|
| 버튼 누름 | 95% 스케일 다운 → 100%로 해제 | 터치 등록 확인 |
| 토글 스위치 | 슬라이드 + 색상 변경 | 상태 변경 가시화 |
| 당겨서 새로고침 | 스피너 등장, 콘텐츠 아래로 시프트 | 액션 처리 중 |
| 폼 제출 | 버튼 → 스피너 → 체크마크 | 진행 → 완료 |
| 에러 | 흔들림 애니메이션 (2-3 사이클, 4px 진폭) | 경고 없이 주의 끌기 |
| 좋아요/즐겨찾기 | 하트 채움 + 스케일 팝 (110% → 100%) | 즐거움의 순간 |

### 규칙
- 마이크로 인터랙션은 <300ms여야 함
- 애니메이션 중에 사용자 입력을 차단하지 마라
- 앱 전체에 일관성
- 시각과 짝지어진 햅틱 피드백 (지원되는 경우)
