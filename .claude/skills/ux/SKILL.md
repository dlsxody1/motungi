## 첫 번째 원칙 (모든 작업)

무언가를 디자인하기 전에 다음 순서대로 답하세요.

1. **사용자의 단 하나의 목표는 무엇인가?** (기능도, 비즈니스 지표도 아닌 — 그들의 실제 의도)
2. **그 목표를 달성하기 위한 최소한의 것은 무엇인가?** (화면, 정보, 행동)
3. **무엇을 제거할 수 있는가?** (제거해도 목표 달성을 막지 않는다면, 제거하라)

**규칙: 1번이 명확하지 않다면, 진행을 멈추고 명확히 하라. `references/design-process.md`의 JTBD 프레이밍을 사용하라.**

## 빠른 진단

무언가가 "잘못됐다"는 느낌이 들거나 개선이 필요할 때:

| 문제 | 주요 원칙 | 조치 | 참고 문서 |
|------|----------|------|----------|
| 사용자가 망설인다 | 힉의 법칙(Hick's Law) | 선택지 줄이기, 점진적 공개 | `cognitive-principles.md` |
| 사용자가 타겟을 놓친다 | 피츠의 법칙(Fitts's Law) | 사이즈 키우고, 거리 줄이기 | `cognitive-principles.md` |
| 사용자가 압도된다 | 인지 부하 | 핵심만 보여주고, 단계로 쪼개기 | `cognitive-principles.md` |
| 사용자가 중도 이탈한다 | 목표 그래디언트 | 진행률 표시, 가시적 마일스톤 | `cognitive-principles.md` |
| 사용자가 부정적으로 기억한다 | 피크-엔드 법칙 | 최악의 순간과 최종 상태 개선 | `cognitive-principles.md` |
| 사용자가 커스텀 UI에 혼란스러워한다 | 야콥의 법칙 | 플랫폼 표준 패턴 사용 | `cognitive-principles.md` |
| 사용자가 콘텐츠를 못 찾는다 | IA 실패 | 내비게이션 재구조화, 트리 테스트로 검증 | `information-architecture.md` |
| 사용자가 무슨 일이 일어났는지 모른다 | 피드백 부재 | 시스템 피드백 추가(토스트, 인라인, 상태 변경) | `interaction-patterns.md` |
| 사용자가 무엇을 해야 할지 모른다 | 카피 실패 | 라벨 재작성, 가이드 추가, 빈 상태 개선 | `ux-writing.md` |
| 시스템이 느리게 느껴진다 | 도허티 임계값 | 낙관적 UI, 스켈레톤 화면, 백그라운드 처리 | `interaction-patterns.md` |
| 3D/XR 이슈 | 전용 참고 문서 참조 | 진단을 위해 적절한 참고 문서를 로드하라 | `3d-design.md` / `xr-design.md` |

## 안티 패턴 (절대 포함하지 말 것)

이들은 "이것이 사용자의 목표 달성을 돕는가?" 테스트를 통과하지 못합니다.

- 제품 플로우 안에 들어간 마케팅 카피, 슬로건, 홍보성 문구
- 앱 화면 안의 모호한 가치 제안이 들어간 히어로 섹션
- 기능적 목적이 없는 장식적 섹션
- 불필요한 온보딩 또는 스플래시 화면 (UI가 자명하다면 생략하라)
- 비파괴적이거나 되돌릴 수 있는 액션에 대한 확인 다이얼로그 (대신 실행 취소를 사용하라)
- "다른 앱에 있으니까" 존재하는 요소들 (첫 번째 원칙 위배)
- 상태를 나타내는 유일한 지표로서의 색상 (접근성 실패)
- 라벨로 사용된 플레이스홀더 텍스트 (포커스 시 사라짐)
- 무엇이 발생할지 명시하지 않은 "정말 하시겠습니까?"
- 표준 작업에 대한 커스텀 패턴 (야콥의 법칙 위배)
- 터치 디바이스에서의 호버 전용 인터랙션
- 조직도 형태의 내비게이션 (사용자 멘탈 모델이 아닌 회사 구조와 일치하는 것)
- 3D/XR 안티 패턴: 도메인별 목록은 `3d-design.md`와 `xr-design.md` 참조

**규칙: "혹시 모르니" 추가하려는 것이라면 — 추가하지 말라. 확신이 없다면, 사용자 테스트 대상으로 표시하라.**

## 핵심 철학

> 가장 좋은 UX는 사용자가 알아채지 못하는 UX다. 사용자는 무엇을 달성했는지 기억해야지, 인터페이스가 어떻게 동작했는지를 기억해서는 안 된다.

- **효율성**: 최소한의 단계, 최소한의 시간, 최소한의 인지 노력
- **명료성**: 화면당 하나의 주요 액션, 명확한 위계, 모호함 없음
- **일관성**: 표준 작업에는 플랫폼 표준 패턴 사용 (야콥의 법칙)
- **관용**: 모든 액션은 되돌릴 수 있거나 비가역 액션 전에 확인을 받는다
- **접근성**: 사후 처리가 아닌 — 처음부터 모든 의사결정에 내장

**규칙: 디자인이 설명을 필요로 한다면, 충분히 명확하지 않은 것이다. 튜토리얼이 필요하다면, 먼저 단순화하라.**

### 8. XR 공간 디자인
안락 영역, 인터랙션 모델(시선, 손, 음성, 컨트롤러), 텍스트 입력, 패스스루 MR 디자인, 콘텐츠 영속성, 공간 UI 패턴, 다중 사용자/소셜 XR, 사이버 멀미 예방, 온보딩, 프라이버시, 플랫폼 디자인 언어(visionOS 26, Quest, Android XR), 접근성.

## 빠른 체크리스트

UX 의사결정을 확정하기 전에:

### 목표 및 구조
- [ ] 사용자의 하나의 목표가 명확히 정의되었다 (JTBD 진술 작성됨)
- [ ] 모든 요소가 "이것이 목표에 도움이 되는가?" 테스트를 통과한다
- [ ] 정보 구조가 검증되었다 (핵심 콘텐츠까지 ≤3 탭)
- [ ] 내비게이션 패턴이 콘텐츠 유형에 적합하다

### 화면 디자인
- [ ] 화면당 주요 액션은 하나이며 시각적으로 우세하다
- [ ] 표시되는 정보는 현재 의사결정에 필요한 것만이다
- [ ] 7가지 화면 상태가 모두 디자인되었다 (빈 상태, 로딩, 로드됨, 에러, 부분, 새로고침, 오프라인)
- [ ] 콘텐츠 위계: 가장 중요한 것이 먼저

### 인터랙션
- [ ] 모든 사용자 액션에 피드백이 존재한다 (시각 + 선택적 햅틱)
- [ ] 파괴적 액션은 확인을 요구하고, 되돌릴 수 있는 액션은 실행 취소를 제공한다
- [ ] 로딩 패턴이 예상 대기 시간과 일치한다
- [ ] 제스처는 가시적인 대안을 가진다

### 카피
- [ ] 버튼 라벨은 구체적인 동사다 ("OK", "제출", "예"가 아닌)
- [ ] 에러 메시지: 무엇이 일어났는지 + 어떻게 고칠지
- [ ] 빈 상태: 설명 + 실행 가능한 CTA
- [ ] 전문 용어 없음, 기술 용어 없음

### 플랫폼 및 인체공학
- [ ] 모바일: 주요 액션은 엄지 영역 안에
- [ ] 안전 영역 준수 (노치, 다이내믹 아일랜드, 홈 인디케이터)
- [ ] 플랫폼 컨벤션 준수 (iOS는 iOS 패턴, Android는 머티리얼)
- [ ] 반응형: 최소 320px에서 테스트됨

### 접근성
- [ ] 명도 대비 비율 통과 (텍스트 4.5:1, UI 컴포넌트 3:1)
- [ ] 색상이 상태의 유일한 지표가 아님
- [ ] 포커스 상태 가시적, 키보드 내비게이션 완전
- [ ] 모든 인터랙티브 요소에 스크린 리더 라벨
- [ ] 다이내믹 타입 / 확장 가능한 텍스트 지원
- [ ] `prefers-reduced-motion` 준수

### 안티 패턴
- [ ] 위 안티 패턴 목록과 대조하여 — 해당 없음 확인
- [ ] 주요 디자인 결정에 대해 인지 원칙이 인용되었음

### 3D / XR (해당하는 경우)
- [ ] `3d-design.md` 및/또는 `xr-design.md`의 검증 체크리스트 실행

## 출력

출력은 두 단계로 분리합니다.

### `docs/ux/ux-design.md` — 애플리케이션 레벨 디자인

앱 전체 UX 의사결정의 단일 진실 원천. 다루는 내용:

- 정보 구조 (사이트맵, 내비게이션 구조)
- 글로벌 내비게이션 패턴 및 라우팅 로직
- 공통 인터랙션 규칙 (토스트, 모달, 에러 처리 컨벤션)
- 공유 상태 패턴 (인증 상태, 오프라인, 로딩)
- 플랫폼별 컨벤션 (iOS vs Android vs 웹)
- 글로벌 적용 접근성 표준
- 페이즈 구현 요약 (페이즈가 존재하는 경우)

각 섹션과 개별 화면 파일로 연결되는 **목차**로 시작하세요.

### `docs/ux/screens/<screen-name>.md` — 화면 레벨 디자인

화면 또는 구분되는 뷰당 한 파일. 각 화면 파일이 다루는 내용:

- 화면 목적 및 사용자 목표 (JTBD)
- 화면 레이아웃 및 콘텐츠 위계
- 7가지 상태 모두 (빈 상태, 로딩, 로드됨, 에러, 부분, 새로고침, 오프라인)
- 화면 내 사용자 플로우 및 의사결정 지점
- 해당 화면 특정의 인터랙션 세부사항
- UX 카피 (라벨, 에러, 빈 상태, 툴팁)
- 페이즈가 존재할 경우 페이즈 태그 (예: `[Phase 1]`)

**네이밍**: 화면 이름과 일치하는 케밥 케이스 사용 (예: `account-settings.md`, `project-dashboard.md`, `onboarding-welcome.md`).
