## 오늘 밤 이슈 선정 배경 (M-002/M-003이 아니라 M-004인 이유)

`docs/backlog/backlog.yml`의 기계적 우선순위 규칙(`status: todo` 중 priority 최상위, 동순위면 id 오름차순)을 그대로 따르면 오늘 밤 집었어야 할 이슈는 **M-002**(목업 데이터 → Supabase 읽기 전환, high)이고, 그다음이 **M-003**(scoring.ts 엣지케이스 테스트 보강, medium)이다.

그러나 planner 단계에서 저장소 상태를 먼저 점검한 결과, M-002와 M-003은 **이미 작업이 완료되어 있었고**, 각각(또는 함께) 브랜치가 만들어져 **open draft PR #6, #7, #8**로 올라가 있는 상태였다. 이 PR들은 오늘(2026-07-13) 기준으로 **3일째(2026-07-10 전후 생성 추정) 사람의 리뷰/머지 없이 정체**되어 있다.

이 상황에서 M-002 또는 M-003을 오늘 밤 다시 집으면 다음과 같은 문제가 생긴다:

- 이미 완료돼 PR로 올라간 작업을 또 다른 브랜치에서 중복 구현하게 되어, 나중에 사람이 PR #6/#7/#8 중 어느 것을 살릴지, 어느 것을 버릴지 판단하는 부담이 늘어난다.
- M-002는 scope가 `apps/web`이라 오늘 새로 만들 브랜치와 파일 경합(예: 같은 컴포넌트를 다른 방식으로 고침) 가능성이 크고, 병합 충돌이 오히려 더 심해진다.
- "밤엔 만들기만, 판단은 아침의 사람이"라는 원칙상, 이미 판단 대기 중인 PR이 3개나 쌓인 상황에서 같은 영역에 또 하나를 쌓는 것은 사람의 검토 부담만 가중시키고 파이프라인의 신뢰도를 떨어뜨린다.

따라서 오늘 밤은 **backlog.yml에서 status가 todo이면서, 아직 어떤 open PR과도 겹치지 않는 다음 이슈**를 찾아 내려갔고, 그 결과 priority가 낮지만(low) 완전히 독립적인 scope(`apps/web/src/app/diagnosis/*`, 신규 테스트 인프라)를 가진 **M-004**를 선택했다. M-004는:

- M-002/M-003과 다루는 화면·로직이 겹치지 않는다(진단 플로우 a11y vs. 데이터 소스 전환/스코어링 로직).
- 저위험·검증 가능(axe 위반 유무, 키보드 포커스 순서는 자동 테스트로 명확히 판정 가능)해서 사람 개입 없이도 안전하게 진행할 수 있다.
- backlog.yml의 M-004 notes에도 원래 "저위험·검증가능 → 야간 파이프라인 첫 대상으로 적합"이라고 명시돼 있었다.

**사람이 반드시 해야 할 조치**: PR #6, #7, #8을 검토해 머지 또는 폐기 결정을 내려야 한다. 이 셋이 계속 열려 있는 한, 다음 야간 실행에서도 M-002/M-003은 계속 스킵될 가능성이 높다(백로그 status가 todo로 남아있는 한 기계적으로는 최우선이지만, 중복 방지 로직이 계속 우회시킬 것이기 때문). 백로그의 status를 실제 PR 상태와 동기화하는 것도 필요하다.
